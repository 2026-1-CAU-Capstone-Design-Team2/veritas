# VERITAS 통합 테스트 시나리오

> AutoSurvey(웹 수집) + 로컬 문서 연결(Local Corpus) + 검증(Verify/Crosscheck) + 초안(Draft) + RAG 채팅 + table_query 를
> **하나의 흐름으로 검증**하기 위한 instruction–내부 데이터 묶음 정의서.

**배경** — VERITAS의 use-case는 "로컬 폴더의 데이터 = 외부로 나가면 안 되는 민감 사내 데이터"를 가정한다.
따라서 실제 사내 데이터를 테스트에 사용할 수 없으며, **민감해 보이는 가상(synthetic) 내부 데이터를 직접 생성**하되
**웹에서 실제로 조사 가능한 주제와 짝을 지어** 전체 파이프라인이 작동하는지 확인한다.

---

## 1. 테스트 설계 원칙

### 1.1 기능별 데이터 요구사항

| 검증 대상 | 데이터 요구사항 |
|---|---|
| **AutoSurvey** | 웹에 공개 자료가 풍부한 실제 주제 (실존 기업/산업/정책) |
| **Crosscheck (검증)** | 내부 데이터의 수치가 웹 공개 수치와 **일부는 일치, 일부는 의도적으로 다르게** 구성 |
| **table_query** | 200행 이상의 CSV/XLSX — 색인 프로필 한계(200행)를 넘는 크기에서 무손실 집계 확인 |
| **RAG 채팅** | ① 웹에만 있는 정보 ② 내부에만 있는 정보 ③ 둘을 합쳐야 답할 수 있는 질문이 모두 성립하도록 |
| **초안 작성 (Draft)** | 내부 + 외부 근거를 함께 인용해야 완성되는 문서 양식 |
| **프라이버시 가드** | "대외비" 성격이 명확한 내부 정보 포함 (OpenAI 가속 ON 상태에서 차단 확인용) |

### 1.2 Crosscheck 작동 방식과 데이터 설계 제약

`services/verification/crosscheck/pipeline.py` 는 다음과 같이 동작한다:

1. 외부(AutoSurvey 수집) 문서의 key_points·요약 문장에서 주장(claim) 추출
2. 로컬 문서의 본문 문장에서 주장 추출
3. **token overlap** 으로 비교쌍 구성 → 수치 불일치(numeric_mismatch)·모순(contradicts) 탐지

따라서:

- 내부 문서는 웹 기사와 **같은 용어**(실존 기업명·제품명·시장 용어)를 사용해야 비교쌍이 만들어진다.
  가상 회사명만 쓰면 crosscheck가 발동하지 않는다.
- 불일치 탐지를 보장하려면 내부 문장이 **"주어 + 수치"** 구조로 웹 문장과 대응되어야 한다.

```
웹 (예상 수집 문장): "삼성전자 DS부문의 2025년 1분기 매출은 23.1조원을 기록했다"
내부 — 일치 설계:    "1분기 DS부문 매출은 23.1조원으로 집계되었다"        → supports
내부 — 불일치 설계:  "내부 집계 기준 1분기 DS부문 매출은 21.5조원이다"     → numeric_mismatch
```

### 1.3 2단계(2-Phase) 데이터 생성 방법론

crosscheck의 기대값("불일치 N건이 탐지되어야 함")을 명확히 하려면 **순서가 중요**하다:

```
Phase 1  AutoSurvey 먼저 실행
         → runs/<workspace>/final.md 와 수집 문서(clean_md/)에서
           웹이 실제로 보고하는 핵심 수치를 5~8개 추출

Phase 2  그 수치를 기준으로 내부 데이터 생성
         → 일치 수치 3~4개 (그대로 복사)
         → 불일치 수치 3~4개 (±10~20% 오프셋)
         → 내부 전용 정보 (웹에 없는 코드명·회의 결정사항 등)

Phase 3  로컬 폴더 등록 → 재색인 → 검증 실행
         → "의도한 불일치 수 = 탐지된 numeric_mismatch 수" 비교
```

> 웹 수치를 미리 알 수 없으므로, 내부 데이터의 구체적인 숫자는 Phase 1 결과를 보고 채워 넣는다.
> 아래 시나리오의 수치는 모두 **예시 형태**이며 Phase 1 이후 실측값 기준으로 치환한다.

## 2. 시나리오 A — 삼성전자 HBM/반도체 사업 (메인 시나리오) ⭐

실존 대기업 + 가상 내부 문서. 웹 자료가 가장 풍부하고 crosscheck가 자연스럽게 발동한다.
**모든 기능을 한 번에 커버하는 메인 시나리오.**

### 2.1 AutoSurvey Instruction

```
삼성전자 2025~2026년 반도체(DS부문) 실적과 HBM 시장 경쟁 구도 조사
```

변형 (특정 출처 강제 포함 테스트):

```
삼성전자 HBM 시장 점유율과 SK하이닉스 경쟁 구도 site:https://news.samsung.com
```

### 2.2 로컬 폴더 구성 (5개 파일, 전부 가상 생성)

```
test_data/scenario_a/
├─ DS부문_내부실적_FY2025.xlsx        # 2시트: 분기별 실적 + 월별 상세(360행)
├─ 고객사별_수주현황_대외비.csv         # 250행
├─ HBM_경쟁사_벤치마크.docx            # 내부 평가 문서
├─ 2026_설비투자_계획초안.md           # 내부 코드명 포함
└─ 임원회의록_0315.txt                # 의사결정 기록
```

### 2.3 파일별 상세 명세

#### ① `DS부문_내부실적_FY2025.xlsx`

**시트 1 — "분기별실적"** (8행: FY2024 Q1 ~ FY2025 Q4)

| 분기 | 매출(조원) | 영업이익(조원) | 영업이익률(%) | 비고 |
|---|---|---|---|---|
| FY2025 Q1 | *(웹 공시값 그대로)* | *(웹 공시값 그대로)* | … | 일치 → supports |
| FY2025 Q2 | *(웹 공시값 그대로)* | … | … | 일치 → supports |
| FY2025 Q3 | *(웹 값 × 0.9)* | *(웹 값 × 0.85)* | … | **불일치 → numeric_mismatch** |
| FY2025 Q4 | *(웹 값 × 1.15)* | … | … | **불일치 → numeric_mismatch** |

**시트 2 — "월별상세"** (360행 = 24개월 × 5사업부 × 3제품군)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| 월 | 텍스트 (`2025-01` 형식) | |
| 사업부 | 텍스트 (메모리·파운드리·LSI·패키징·R&D) | group_by 테스트용 |
| 제품군 | 텍스트 (HBM·DRAM·NAND) | 필터 테스트용 |
| 매출(억원) | 숫자 — `"1,234"` 콤마 표기 | 숫자 코어싱 테스트용 |
| 수출비중(%) | 숫자 | mean 집계 테스트용 |

> **검산 기대값을 파일 생성 시점에 기록**해 둔다 (예: 전체 매출 합계, HBM 제품군 합계, 사업부별 합계).
> table_query 결과와 이 기대값을 대조하는 것이 무손실 검증의 핵심이다.

#### ② `고객사별_수주현황_대외비.csv` (250행)

| 컬럼 | 타입 | 비고 |
|---|---|---|
| 고객사명 | 텍스트 (가상: "노스스타시스템즈", "블루오션테크", "퀀텀리프AI" 등 20개사) | 완전 내부 정보 |
| 지역 | 텍스트 (북미·유럽·중국·국내·동남아) | group_by 테스트용 |
| 제품 | 텍스트 (HBM3E·DDR5·LPDDR5X 등) | |
| 수주액(억원) | 숫자 — `"45,000"` 콤마 표기 | sum/상위 N 테스트용 |
| 수주일 | 날짜 (`2025-03-15` ISO 형식) | 날짜 필터 테스트용 |
| 납기일 | 날짜 | |
| 상태 | 텍스트 (진행중·완료·보류) | 필터 테스트용 |

#### ③ `HBM_경쟁사_벤치마크.docx`

내부 평가 문서 형식. **웹과 같은 용어, 일부 다른 수치** 포함:

```
1. HBM 시장 개요
   - "2026년 글로벌 HBM 시장 규모는 ___억 달러로 전망된다"   ← 웹 수치 그대로 (supports)
   - "당사 내부 추정으로는 ___억 달러 수준"                  ← 웹 값 × 0.8 (mismatch)
2. 경쟁사 분석
   - SK하이닉스 점유율: 웹 수치와 다르게 기재                 ← mismatch
   - 마이크론 점유율: 웹 수치 그대로                          ← supports
3. 당사 포지션 (완전 내부 정보)
   - 수율 현황, 고객 인증 진행 상황, 기술 격차 자체 평가
```

#### ④ `2026_설비투자_계획초안.md`

웹에 존재하지 않는 **내부 전용 정보**로만 구성 (RAG 내부 전용 질문의 근거):

```
# 2026년 설비투자 계획 (초안) — 대외비

## 프로젝트 아폴로 (평택 P5 증설)
- 투자 우선순위 1위, 예산 12.5조원
## 프로젝트 헬리오스 (HBM4 전용 라인)
- 투자 우선순위 2위, 예산 8.2조원
## 보류 항목
- 미국 테일러 2공장 추가 투자는 2027년으로 연기 결정
```

#### ⑤ `임원회의록_0315.txt`

```
2026-03-15 DS부문 경영전략회의 회의록 (대외비)
참석: ...(가상 인물)
결정사항:
1. HBM4 양산 일정을 2026년 4분기에서 3분기로 앞당김
2. 프로젝트 아폴로 예산 12.5조원 승인
3. 중국 시안 공장 NAND 라인 전환은 보류
리스크 논의:
- 경쟁사 대비 수율 격차 해소가 최우선 과제
```

### 2.4 테스트 질문 셋 및 기대 결과

| # | 분류 | 질문 | 기대 동작 | 합격 기준 |
|---|---|---|---|---|
| A-1 | RAG (웹) | "HBM 시장에서 삼성전자의 경쟁 위치는 어때?" | `rag_search` 발동, 웹 출처 인용 | 수집 문서 기반 답변 + 출처 표기 |
| A-2 | RAG (내부) | "임원회의에서 결정된 투자 우선순위가 뭐였지?" | `rag_search` 발동, 로컬 출처 인용 | "프로젝트 아폴로 1위" 등 내부 정보로 답변 |
| A-3 | RAG (혼합) | "웹에서 조사된 HBM 시장 전망과 우리 내부 벤치마크 평가를 비교해줘" | 웹 + 로컬 양쪽 근거 인용 | 두 출처가 모두 인용되고 차이가 언급됨 |
| A-4 | table_query | "고객사별 수주 총액 상위 5개 보여줘" | `table_query` 발동 (rag_search 아님) | group_by + sum + sort 결과가 검산값과 일치 |
| A-5 | table_query | "월별상세 시트에서 HBM 제품군 매출 합계가 얼마야?" | `table_query` 발동, 시트 지정 | **360행 전체** 기준 합계 = 검산값 (200행 제한 없음 증명) |
| A-6 | table_query | "수주 상태가 진행중인 건수는?" | `table_query` count | 검산값과 일치 |
| A-7 | 검증 | 검증 페이지에서 정합성 검증 실행 | crosscheck 파이프라인 실행 | 의도한 불일치(4건 내외)가 **"Cross-check 결과" 카드**에 내부 주장 vs 외부 주장 쌍으로 표시됨 |
| A-8 | 초안 | "DS부문 2026년 사업계획서 초안 작성" (양식: 사업 제안서) | KnowledgePackBuilder가 웹+로컬 근거 수집 | 초안에 내부·외부 근거가 모두 인용됨 |
| A-9 | 프라이버시 | OpenAI 가속 ON + A-2 질문 재실행 | 로컬 데이터 → 외부 LLM 차단 | 거부 동작 확인 (채팅은 로컬 LLM이므로 정상 동작, draft에서 409 확인) |
| A-10 | 증분 색인 | XLSX 수치 1개 수정 후 재색인 | content_hash 변경 감지 | "신규 1, 건너뜀 4" 표시 |

## 3. 시나리오 B — 대체육 시장 + 가상 식품기업 (스모크 테스트)

내부 데이터가 단순해서 **빠른 동작 확인**에 적합. 기능 수정 후 회귀 확인용.

### 3.1 AutoSurvey Instruction

```
국내외 대체육(식물성 단백질) 시장 규모와 성장 전망, 주요 업체 동향 조사
```

### 3.2 로컬 폴더 구성

```
test_data/scenario_b/
├─ 월별_제품매출.csv          # 300행: 월 × 제품 × 채널
├─ 신제품_출시계획.docx        # "(주)그린테이블" 가상 회사의 출시 로드맵
└─ 소비자패널_조사결과.md      # 가상 소비자 조사 요약
```

### 3.3 Crosscheck 주의사항

가상 회사명을 쓰므로 회사 실적 문장으로는 crosscheck가 발동하지 않는다.
대신 **시장 수치 문장**을 내부 문서에 의도적으로 포함시킨다:

```
소비자패널_조사결과.md 에 포함할 문장 예시:
- "국내 대체육 시장 규모는 ___억원으로 추정된다"      ← Phase 1에서 확인한 웹 수치 ±20%
- "글로벌 대체육 시장은 2030년 ___억 달러 전망"       ← 웹 수치 그대로
```

### 3.4 테스트 질문 셋

| # | 분류 | 질문 | 합격 기준 |
|---|---|---|---|
| B-1 | RAG (웹) | "국내 대체육 시장의 주요 업체는?" | 웹 수집 문서 기반 답변 |
| B-2 | table_query | "채널별 매출 합계 비교해줘" | 300행 전체 group_by 집계 일치 |
| B-3 | 검증 | 정합성 검증 실행 | 시장 규모 불일치 1건 이상 탐지 |
| B-4 | 초안 | "신제품 출시 보고서 초안 작성" | 내부 로드맵 + 웹 시장 자료 모두 인용 |

## 4. 시나리오 C — 지자체 탄소중립 정책 (table_query 집중)

공공 주제라 웹 자료가 풍부하고, **대용량 수치 데이터의 table_query 정밀 테스트**에 최적.

### 4.1 AutoSurvey Instruction

```
국내 지자체 탄소중립 정책 사례와 예산 규모, 온실가스 감축 목표 조사
```

### 4.2 로컬 폴더 구성

```
test_data/scenario_c/
├─ 부서별_에너지사용량.xlsx     # 500행: 월 × 부서 × 에너지원 (전기/가스/지역난방)
├─ 정책검토_내부보고서.docx     # 타 지자체 사례 대비 자체 평가 (수치 일부 불일치 설계)
└─ 예산편성_초안.md            # 내부 예산 배분 계획
```

## 6. 기능별 합격 기준 요약

| 기능 | 합격 기준 |
|---|---|
| **AutoSurvey** | final.md 생성, 수집 문서 5건 이상, 진행률 실시간 표시 |
| **로컬 색인** | 등록 파일 전부 "indexed", 실패 0건, 출처 목록에 표시 |
| **RAG (웹/내부/혼합)** | 질문 유형에 맞는 출처가 인용됨. 내부 전용 질문에 웹 지식으로 답하지 않음 |
| **table_query** | ① rag_search가 아닌 table_query가 발동 ② 200행 초과 데이터의 집계가 검산값과 일치 ③ 콤마 표기 수치가 올바르게 계산됨 |
| **Crosscheck** | 의도적으로 설계한 불일치가 numeric_mismatch/contradicts로 탐지됨 (재현율 기록) |
| **초안** | 내부 + 외부 근거가 모두 source map에 포함되고 본문에 인용됨 |
| **프라이버시 가드** | OpenAI 가속 ON 상태에서 로컬 데이터가 외부로 전송되지 않음 (채팅은 로컬 LLM 유지, draft는 409) |
| **증분 색인** | 수정된 파일만 재색인 ("건너뜀" 카운트 확인) |

## 부록 A. 합성 데이터 생성 방법

수동 작성 대신 스크립트로 생성을 권장한다 (검산 기대값을 자동으로 함께 산출하기 위함):

```
tests/fixtures/generate_scenario_data.py (작성 예정)
  - openpyxl  → .xlsx 생성 (시트 2개, 360행/500행)
  - csv       → .csv 생성 (250행/300행, 콤마 표기 수치)
  - python-docx → .docx 생성
  - 표준 출력  → 검산 기대값 (전체 합계, group별 합계, 상위 N) JSON 출력
```

생성 시 주의:

1. 수치에 콤마 표기(`"45,000"`)와 단위 접미사(`"원"`, `"%"`)를 섞어 숫자 코어싱을 함께 검증
2. 날짜는 ISO 형식(`2025-03-15`)으로 통일 (문자열 비교 정렬이 올바르게 동작)
3. 빈 셀을 일부 포함시켜 null 처리 검증
4. 시트/컬럼명에 한국어 사용 (한국어 컬럼 매칭 검증)
